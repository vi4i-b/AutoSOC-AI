# ТЗ для разработчика: багфиксы и антифишинговая система

## Цель документа

Этот документ описывает цели, задачи и критерии готовности для основного разработчика AutoSOC.
Приоритеты разделены так, чтобы сначала стабилизировать продукт, затем закрыть критичные баги,
а после этого внедрить полноценную антифишинговую подсистему.

---

## Главная цель

Довести AutoSOC до состояния стабильного desktop-продукта, в котором:

- нет критических багов, мешающих логину, сканированию, Telegram-уведомлениям и firewall-действиям;
- все основные подсистемы ведут себя предсказуемо и логируются;
- реализована встроенная антифишинговая система;
- продукт можно безопасно собирать, тестировать и передавать пользователям.

---

## Блок 1. Полное закрытие критичных и средних багов

### Цель

Устранить все баги, влияющие на стабильность, безопасность и доверие к продукту.

### Задачи

1. Провести полный bug audit по модулям:
   - `login.py`
   - `auth.py`
   - `database.py`
   - `main.py`
   - `scanner.py`
   - `guard.py`
   - `log_listener.py`
   - `ai_expert.py`

2. Собрать баг-лист по категориям:
   - логин и регистрация;
   - Telegram binding;
   - сканирование и обработка результатов;
   - firewall-операции;
   - real-time detection;
   - AI panel;
   - сборка `.exe`;
   - хранение состояния и recovery после перезапуска.

3. Закрыть P0 баги:
   - падения приложения;
   - зависания UI;
   - неработающий login/register;
   - поломка доступа к базе;
   - неверные firewall-действия;
   - ошибки, приводящие к ложной блокировке порта или IP;
   - баги, из-за которых scan/alert не завершается.

4. Закрыть P1 баги:
   - некорректные статусы Telegram;
   - рассинхрон UI и реального состояния firewall;
   - сбой сохранения настроек;
   - плохая обработка ошибок внешних зависимостей;
   - потеря baseline между сессиями;
   - баги локализации и кодировки в UI/документации.

5. Закрыть P2 баги:
   - неточные тексты ошибок;
   - визуальные дефекты;
   - несогласованные формулировки;
   - неочевидные состояния интерфейса;
   - остаточные проблемы упаковки и portable-режима.

### Критерии готовности

- Нет crash-багов на основных сценариях.
- Все ключевые действия дают понятный статус пользователю.
- Ошибки логируются, а не теряются silently.
- Есть таблица известных багов с пометкой `fixed / verified / deferred`.

## Блок 2. Стабилизация архитектуры и технического долга

### Цель

Упростить сопровождение проекта и убрать архитектурные риски, из-за которых баги будут возвращаться.

### Задачи

1. Уменьшить перегруженность `main.py`:
   - вынести firewall-операции в отдельный сервис;
   - вынести scan orchestration в отдельный сервис;
   - вынести Telegram operations в отдельный сервис;
   - вынести incident/audit formatting в отдельный сервис.

2. Зафиксировать единую схему БД:
   - описать структуру таблиц;
   - убрать неявные расхождения между старой и новой схемой;
   - документировать миграции;
   - проверить апгрейд старых `.db`.

3. Навести порядок в конфиге:
   - единый способ чтения `.env`;
   - централизованный runtime config;
   - валидация обязательных параметров;
   - диагностический вывод о недостающих зависимостях.

4. Усилить журналирование:
   - security events;
   - audit events;
   - build/runtime errors;
   - failed integrations;
   - phishing verdict history.

5. Внедрить централизованную обработку ошибок:
   - безопасные fallback-ветки;
   - user-friendly сообщения;
   - технические детали в логах;
   - отсутствие пустых `except`.

### Критерии готовности

- Основные сервисы отделены от UI.
- Старые базы открываются без потери данных.
- Конфигурация диагностируется автоматически.
- В коде нет критичных мест с неуправляемыми side effects.

## Блок 3. Качество, тестирование и релизный контур

### Цель

Сделать разработку предсказуемой, а релизы воспроизводимыми.

### Задачи

1. Добавить тесты:
   - unit tests на auth/database/validators/analyzer;
   - smoke tests на scan flow;
   - tests на Telegram integration через mocks;
   - tests на phishing detector;
   - tests на migration compatibility.

2. Добавить check pipeline:
   - `compileall`;
   - `unittest` или `pytest`;
   - lint;
   - базовая проверка build.

3. Проверить релизный сценарий:
   - чистая сборка `.exe`;
   - запуск на машине без dev-окружения;
   - проверка `Nmap`/`Npcap`/Windows Event Log;
   - portable mode;
   - installer scenario.

4. Упорядочить runtime-файлы:
   - отделить dev artifacts от release artifacts;
   - не хранить рабочие базы и временные файлы в git;
   - подготовить release checklist.

### Критерии готовности

- Есть минимальный regression suite.
- Сборка повторяема и понятна другому разработчику.
- Основные сценарии можно прогнать по чек-листу.

## Блок 4. Реализация антифишинговой системы

### Цель

Добавить в AutoSOC полноценный anti-phishing модуль, который анализирует подозрительные письма,
URL, домены и вложения, выдает риск-оценку и рекомендации пользователю.

### Целевой результат

В приложении должен появиться отдельный anti-phishing workflow:

- пользователь вставляет URL, текст письма, заголовки письма или домен;
- система анализирует артефакт;
- формируется verdict: `safe / suspicious / malicious / unknown`;
- отображаются причины verdict;
- можно отправить результат в Telegram и сохранить в журнал.

### Задачи по архитектуре

1. Создать отдельный модуль, например:
   - `phishing_detector.py`
   - `phishing_rules.py`
   - `phishing_ui.py` или встроенную вкладку в `main.py`

2. Спроектировать модель данных:
   - `artifact_type`
   - `source_text`
   - `normalized_url`
   - `domain`
   - `brand_impersonation`
   - `risk_score`
   - `verdict`
   - `signals`
   - `recommendations`
   - `created_at`

3. Добавить таблицу/хранилище:
   - история phishing-анализов;
   - verdict;
   - signal list;
   - user action;
   - экспорт и журнал.

### Задачи по детекции

1. Реализовать URL-анализ:
   - нормализация URL;
   - выделение домена и subdomain;
   - детекция IP вместо домена;
   - punycode/IDN detection;
   - слишком длинные URL;
   - excessive redirects preparation;
   - suspicious query parameters;
   - URL shortener detection;
   - lookalike domains;
   - brand spoofing heuristics.

2. Реализовать анализ текста письма:
   - urgent language detection;
   - credential harvesting patterns;
   - fake invoice/payment triggers;
   - password reset scams;
   - attachment lures;
   - просьбы срочно перейти по ссылке;
   - language mismatch;
   - generic greeting detection.

3. Реализовать анализ email headers:
   - From / Reply-To mismatch;
   - SPF/DKIM/DMARC fields parsing, если доступны;
   - подозрительные relay hops;
   - несогласованность sender identity.

4. Реализовать анализ вложений на первом этапе без sandbox:
   - suspicious extensions;
   - double extension;
   - macro-enabled Office files;
   - archive-with-executable heuristic;
   - embedded script patterns by filename.

5. Построить scoring engine:
   - вес каждого сигнала;
   - пороги `safe / suspicious / malicious`;
   - explainable verdict;
   - ручная калибровка через конфиг.

### Задачи по UX

1. Добавить отдельный UI-блок:
   - поле для URL;
   - поле для email text;
   - поле для headers;
   - drag-and-drop или выбор файла;
   - кнопка `Analyze`.

2. Показать результат человеку понятным образом:
   - общий риск score;
   - итоговый verdict;
   - список сработавших сигналов;
   - почему это подозрительно;
   - что делать дальше.

3. Добавить кнопки действий:
   - отправить в Telegram;
   - сохранить в журнал;
   - экспортировать отчёт;
   - передать в AI analysis.

### Задачи по интеграции

1. Интегрировать anti-phishing модуль с AI assistant.
2. Интегрировать результаты в audit/security events.
3. Добавить шаблон Telegram alert для phishing verdict.
4. Добавить историю anti-phishing проверок в UI.

### Критерии готовности

- Можно вставить URL и получить объяснимый verdict.
- Можно вставить текст письма и получить phishing risk analysis.
- Можно сохранить и повторно открыть результаты анализа.
- Verdict не является только числом, а объясняется сигналами.
- Антифишинговый модуль покрыт тестами.
